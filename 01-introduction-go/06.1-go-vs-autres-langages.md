# 1.6.1 Go vs Rust / Python / Java / C# : quand choisir quoi

La [grille générale](06-positionnement-2026.md) raisonne par cas d'usage ; cette section compare Go directement à ses quatre rivaux les plus fréquents. Précision de méthode : ces quatre langages sont **excellents et matures**. La question n'est jamais « lequel est le meilleur », mais « lequel convient à *ce* problème, à *cette* équipe, sous *ces* contraintes ». Les comparaisons ci-dessous décrivent des **tendances, pas des lois**.

*(Pour la correspondance de syntaxe, voir l'[annexe A](../annexes/correspondance-go-autres/README.md) ; pour migrer un service existant, la [section 11.3](../11-interop-migration/03-migrer-vers-go.md).)*

## Go vs Rust — simplicité contre contrôle

Rust est, comme Go, compilé, typé statiquement et produit des binaires natifs — mais **sans ramasse-miettes**. Sa sûreté mémoire repose sur un système de possession et d'emprunt (*ownership/borrowing*), avec des abstractions à coût nul, une performance et un contrôle maximaux, et une concurrence « sans peur ». La contrepartie : une courbe d'apprentissage raide, des compilations plus lentes et une complexité supérieure.

- **Choisir Go quand** : services, API, CLI, cloud-native ; la vélocité et la simplicité priment ; le ramasse-miettes est acceptable ; l'équipe doit être productive rapidement.
- **Choisir Rust quand** : le GC n'est pas tolérable (temps réel, latence déterministe), contrôle mémoire fin, embarqué, performance de pointe, briques système (OS, moteurs, navigateurs), ou sûreté mémoire *sans* GC.

*En un mot* : Go optimise le temps du développeur ; Rust, le contrôle et la performance de pointe. Les deux progressent — le ramasse-miettes et les performances de Go se sont affinés (voir [module 14](../14-performance/README.md)).

## Go vs Python — production contre prototypage

Python est dynamique, interprété, et d'une productivité redoutable pour scripter et prototyper. Il règne surtout sur la **data science, le ML et le calcul scientifique** (NumPy, pandas, PyTorch…). Ses revers : une exécution plus lente, un GIL qui limite le parallélisme CPU, un typage dynamique qui repousse des erreurs à l'exécution, et un déploiement plus lourd (interpréteur et dépendances).

- **Choisir Go quand** : services de production à fort trafic, concurrence, performance et faible mémoire, sûreté de typage à la compilation, déploiement en binaire unique — typiquement lorsqu'on a « dépassé » les performances de Python.
- **Choisir Python quand** : data science, ML, calcul scientifique, scripts, prototypage rapide, ou quand l'écosystème de bibliothèques est décisif.

Un motif fréquent : prototyper en Python, puis réécrire en Go les services critiques pour la performance (voir [11.3](../11-interop-migration/03-migrer-vers-go.md)) — et souvent, Go pour l'infrastructure *autour* d'un cœur ML resté en Python.

*Nuance 2026* : CPython gagne un mode **« *free-threaded* » sans GIL**, officiellement pris en charge depuis Python 3.14 (PEP 779) mais encore **optionnel** — ce n'est pas le build par défaut. Le verrou global s'estompe donc progressivement, sans avoir encore disparu : en pratique courante, le parallélisme CPU de Python reste contraint.

*En un mot* : Python pour explorer et pour la data ; Go pour tenir la charge en production.

## Go vs Java — légèreté contre écosystème

Java est très mature, doté d'un écosystème colossal et d'une JVM performante (JIT efficace à chaud, GC éprouvés), avec des frameworks d'entreprise solides (Spring) et un outillage riche. Ses revers historiques — empreinte de la JVM au démarrage et en mémoire, verbosité — sont atténués par les images natives (GraalVM) et les évolutions récentes du langage.

- **Choisir Go quand** : microservices cloud-native, CLI, démarrage rapide, faible mémoire, petits conteneurs, simplicité ; projets neufs sans dette JVM.
- **Choisir Java quand** : gros systèmes d'entreprise avec un existant Java, besoin du vaste écosystème JVM, équipes déjà sur la JVM, domaines où des frameworks matures sont décisifs, ou débit maximal sur des services longue durée.

*Nuance 2026* : les *virtual threads* (Project Loom, stables depuis Java 21) rapprochent l'ergonomie de concurrence de la JVM de celle des goroutines.

*En un mot* : Go pour la légèreté et le déploiement ; Java pour la profondeur d'écosystème et l'entreprise.

## Go vs C# — minimalisme contre polyvalence

C# est un langage moderne et riche (souvent plus que Go), porté par la plateforme **.NET** désormais multiplateforme, un outillage de premier ordre et un modèle `async`/`await` mûr. ASP.NET Core en fait une pile web très performante, et C# brille par sa polyvalence : desktop (WinUI, MAUI), jeu vidéo (Unity), applications d'entreprise.

- **Choisir Go quand** : cloud-native, CLI, services épurés, empreinte minimale, binaire statique, simplicité du modèle de concurrence.
- **Choisir C# quand** : applications d'entreprise (Windows ou multiplateformes), besoin de l'écosystème .NET, interfaces graphiques de bureau, jeu vidéo (Unity), équipes déjà sur .NET, ou pile web ASP.NET Core.

*Nuance 2026* : NativeAOT (.NET) réduit l'écart sur le binaire compact et le démarrage rapide.

*En un mot* : Go pour le minimalisme et l'infrastructure ; C# pour la polyvalence et l'étendue applicative.

## Récapitulatif

| Face à… | Préférer Go quand… | Préférer l'autre quand… |
|---------|--------------------|-------------------------|
| **Rust** | vélocité, simplicité, GC acceptable | pas de GC, temps réel, contrôle mémoire, embarqué, perf de pointe |
| **Python** | production, concurrence, perf, typage, déploiement | data science / ML, scripts, prototypage, écosystème |
| **Java** | légèreté, démarrage, petits conteneurs, projet neuf | existant JVM, écosystème d'entreprise, débit longue durée |
| **C#** | minimalisme, binaire unique, infrastructure | écosystème .NET, desktop / GUI, Unity, polyvalence |

## À retenir

- Ces comparaisons décrivent des **tendances**, pas des règles : le contexte (équipe, existant, contraintes) tranche souvent plus que le langage lui-même.
- Beaucoup d'objections classiques — contre Go comme contre ses rivaux — se sont **atténuées** ; mieux vaut raisonner sur l'état de l'art 2026 que sur des souvenirs.
- Le but n'est pas d'élire un vainqueur, mais de **choisir l'outil adapté** — et rien n'empêche de faire cohabiter plusieurs langages dans un même système.

## En résumé

Face à Rust, Go échange le contrôle absolu contre la simplicité ; face à Python, la souplesse d'exploration contre la robustesse en production ; face à Java et C#, l'empreinte de plateformes riches contre la légèreté d'un binaire autonome. Dans chaque cas, Go se distingue moins par une supériorité brute que par un **compromis assumé** en faveur de la simplicité et du déploiement. Reste, pour clore ce module, à choisir non plus un langage mais un environnement de travail : [GoLand ou VS Code](06.2-goland-vs-vscode.md).
